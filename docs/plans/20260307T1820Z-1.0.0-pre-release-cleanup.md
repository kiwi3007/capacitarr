## Phase 1: Make In-App Notifications Always-On ✅

Currently, in-app notifications only work if the user manually creates a notification channel of type `"inapp"` and toggles subscriptions. This is confusing — in-app should be a built-in system feature that is always active. All changes in this phase must strictly follow the service layer architecture: no DB access from route handlers, all business logic in services, proper event publishing, and full test coverage.

### Step 1.1 — Update NotificationDispatchService to always write in-app ✅

**Files:** `backend/internal/services/notification_dispatch.go`

- ✅ In `dispatchDigest()`: After the channel loop, unconditionally call `s.senders["inapp"].SendDigest("", digest)` so every digest is persisted in-app regardless of channel configuration. Log errors but do not fail the overall dispatch.
- ✅ In `dispatchAlert()`: After the channel loop, unconditionally call `s.senders["inapp"].SendAlert("", alert)` so every alert is persisted in-app.
- ✅ Guard with a nil check on the sender (defensive programming — uses map lookup `ok` check).
- ✅ Publish `NotificationSentEvent` for in-app sends with `ChannelType: "inapp"` and appropriate `TriggerType`.

### Step 1.2 — Remove "inapp" as a configurable channel type ✅

**Files:** `backend/internal/db/validation.go`, `backend/routes/notifications.go`

- ✅ Remove `"inapp": true` from `ValidNotificationChannelTypes` in `validation.go`.
- ✅ Update the validation error message in `notifications.go` POST handler: "type must be discord or slack".
- ✅ Update the PUT handler similarly.
- ✅ Removed unused `notifTypeInApp` constant from `routes/notifications.go`.

### Step 1.3 — Add database migration to clean up existing inapp channels ✅

**File:** `backend/internal/db/migrations/00005_remove_inapp_channel_type.sql`

- ✅ Add a new migration that deletes any `notification_configs` rows where `type = 'inapp'`.
- ✅ This is a one-way migration — users won't lose anything since in-app is now automatic.
- **Note:** Used `-- +goose Up` / `-- +goose Down` format (not `-- +migrate Up`) to match existing migration conventions.

### Step 1.4 — Skip inapp channels in dispatch loops ✅

**File:** `backend/internal/services/notification_dispatch.go`

- ✅ In `dispatchDigest()` and `dispatchAlert()`, add a `continue` if `cfg.Type == "inapp"` in the channel loop, since in-app is now handled unconditionally outside the loop. This prevents duplicate in-app notifications if somehow a legacy config still exists.

### Step 1.5 — Update tests ✅

**File:** `backend/internal/services/notification_dispatch_test.go`

- ✅ Add test verifying in-app notifications fire even with zero enabled channels (`TestNotificationDispatch_InAppDigestWithZeroChannels`).
- ✅ Add test verifying in-app notifications fire alongside Discord/Slack channels (`TestNotificationDispatch_InAppDigestAlongsideExternal`).
- ✅ Add test verifying in-app alerts fire with zero enabled channels (`TestNotificationDispatch_InAppAlertWithZeroChannels`).
- ✅ Refactored `newTestDispatch` to use separate mocks for external channels (discord/slack) and in-app, returning both. Updated all 11 existing callers.
- ✅ Tests use canonical "Serenity" media name where applicable.

### Step 1.6 — Frontend: remove inapp from channel type dropdown ✅

**Files:** `frontend/app/components/settings/SettingsNotifications.vue`, `frontend/app/types/api.ts`

- ✅ Removed "In-App" `<UiSelectItem>` from the notification channel creation dropdown.

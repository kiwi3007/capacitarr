## Coding Rules & Guidelines

These rules apply to **all phases** of 2.0 development. They supplement the project-wide rules in `.kilocoderules`.

### Backend Rules

1. **Service layer architecture is mandatory.** All DB access goes through services. No exceptions. See `.kilocoderules` — "Service Layer Architecture" section.
2. **Every new service** must accept `*gorm.DB` and `*events.EventBus`, be registered on `services.Registry`, and include unit tests using the in-memory SQLite pattern.
3. **New integration code** must implement capability interfaces (see Phase 2), not the old monolithic `Integration` interface.
4. **All new scoring factors** must implement `ScoringFactor` interface (see Phase 2). No hardcoded scoring logic in the evaluator loop.
5. **All new enrichers** must implement the `Enricher` interface (see Phase 2). No additions to a monolithic `EnrichItems()` function.
6. **Conventional Commits** for every commit. Format: `<type>(<scope>): <description>`.
7. **`make ci` must pass** before any commit is declared ready.
8. **Canonical test names:** Use "Firefly" for TV shows, "Serenity" for movies in all test data.
9. **No `!important` in CSS.** Fix the root cause. No `eslint-disable` or `@ts-ignore`. No warning suppression.
10. **Use shadcn-vue components** everywhere. No raw `<input>`, `<select>`, `<table>`, etc.

### Frontend Rules

1. **ECharts** for all new charts (Phase 3 migrates existing ApexCharts). Tree-shake to keep bundle size reasonable.
2. **`@tanstack/vue-virtual`** for any list that could exceed ~50 items.
3. **`DashboardCard`** component for every analytics card — no one-off card wrappers.
4. **Granular SSE subscriptions** — components subscribe to specific event types, not `*`.
5. **Query parameter pattern** for cross-page linking: `/library?filter=dead`, `/library?quality=720p`, `/library?ruleId=5`.
6. **i18n** — every user-visible string must use the i18n system, even for new features. Add keys to all 22 language files (English values as placeholder for untranslated strings is acceptable).

### Schema Rules

1. **No incremental migrations from 1.x.** The 2.0 schema is a clean baseline.
2. **All new tables and columns** follow the naming conventions established in the baseline migration.
3. **Foreign keys** are required for all relationships. Cascading deletes where appropriate.
4. **No nullable columns without a documented reason.** Prefer `NOT NULL` with defaults.

## Phase 1: Fresh Schema Baseline

**Goal:** Replace the 11 incremental migration files with a single new baseline that reflects the 2.0 domain model.

**Design reference:** [Original plan — Fresh Schema Baseline](20260318T2119Z-capacitarr-2.0-plan.md#fresh-schema-baseline)

### Steps

1. **Design the 2.0 schema.** Document all tables, columns, types, constraints, and relationships in a schema design doc or directly in the migration SQL file with comments. Key new entities:
   - `libraries` — first-class library entity (see [Library Abstraction](20260318T2119Z-capacitarr-2.0-plan.md#7-library-abstraction))
   - Renamed integration type: `overseerr` → `seerr` (see [Overseerr → Seerr Rename](20260318T2119Z-capacitarr-2.0-plan.md#overseerr--seerr-rename))
   - Nullable `threshold_pct` / `target_pct` on `integration_configs` for per-integration thresholds
   - Pluggable scoring factor weight columns
2. **Delete existing migration files** (`00001_baseline.sql` through `00011_add_cancelled_audit_action.sql`).
3. **Create `00001_v2_baseline.sql`** — single comprehensive migration file.
4. **Update `db/models.go`** to match the new schema. All GORM model structs must align with the baseline SQL.
5. **Update `db/migrate.go`** to reference only the new baseline.
6. **Update `db/validation.go`** if any validation logic needs adjustment for new/changed columns.
7. **Run `make ci`** — all existing tests will break here; that's expected. Fix model-level test compilation errors only (integration-level test fixes happen in later phases).

### Acceptance Criteria

- [x] Single `00001_v2_baseline.sql` migration file exists
- [x] All old migration files removed
- [x] `db/models.go` struct definitions match the baseline schema exactly
- [x] `libraries` table present with `id`, `name`, `disk_group_id`, `threshold_pct`, `target_pct` columns
- [x] `integration_configs.type` accepts `"seerr"` (no `"overseerr"`)
- [x] `integration_configs` has nullable `threshold_pct` and `target_pct` columns
- [x] Schema design is documented (either inline SQL comments or a separate doc)
- [x] `make ci` passes (model-level tests compile and pass)

> **Note:** `db/migrate.go` did not need changes — it uses `embed.FS` to discover migration files by glob, so the renamed file is picked up automatically. Also, `db/db.go` was updated to seed the new preference fields (`RequestPopularityWeight`, `QualityBloatWeight`, `DeadContentMinDays`, `StaleContentDays`). All existing tests pass without modification because the schema changes are additive (new tables, new nullable columns on existing tables).

## Phase 2: Backend Architectural Refactors

**Goal:** Restructure the backend internals to use capability-based interfaces, a composable enrichment pipeline, pluggable scoring factors, and an extracted evaluator. This phase is pure refactoring — no new features, no new API endpoints.

**Design reference:** [Original plan — Architectural Refactors](20260318T2119Z-capacitarr-2.0-plan.md#architectural-refactors)

### Steps

1. **Define capability interfaces** in `internal/integrations/types.go`:
   - `Connectable`, `MediaSource`, `DiskReporter`, `MediaDeleter`, `WatchDataProvider`, `RequestProvider`, `WatchlistProvider`
   - See [original plan — §1 Capability-Based Integration Interfaces](20260318T2119Z-capacitarr-2.0-plan.md#1-capability-based-integration-interfaces) for exact signatures
2. **Refactor each integration client** to implement only the interfaces it supports:
   - Rename `overseerr.go` → `seerr.go`, `OverseerrClient` → `SeerrClient`, constant `IntegrationTypeOverseerr` → `IntegrationTypeSeerr`
   - Sonarr: `Connectable + MediaSource + DiskReporter + MediaDeleter`
   - Radarr: same as Sonarr
   - Readarr, Lidarr: same pattern, scope per actual capabilities
   - Plex: `Connectable + WatchDataProvider + WatchlistProvider`
   - Tautulli: `Connectable + WatchDataProvider`
   - Seerr: `Connectable + RequestProvider`
   - Jellyfin: `Connectable + WatchDataProvider + WatchlistProvider`
   - Emby: `Connectable + WatchDataProvider + WatchlistProvider`
3. **Build the `IntegrationRegistry`** in `internal/integrations/registry.go`:
   - Stores registrations keyed by integration ID
   - Methods: `WatchProviders()`, `MediaSources()`, `DiskReporters()`, `RequestProviders()`, `Deleters()`, etc.
   - See [original plan — §2 Registry-Based Integration Discovery](20260318T2119Z-capacitarr-2.0-plan.md#2-registry-based-integration-discovery)
4. **Implement plugin-style registration** with factory functions:
   - `Register("sonarr", SonarrFactory{})` pattern in `init()` or an explicit `RegisterAll()` call
   - See [original plan — Plugin-Style Integration Registration](20260318T2119Z-capacitarr-2.0-plan.md#plugin-style-integration-registration)
5. **Refactor the `MediaItem` model** to the layered structure:
   - Core `MediaItem` + `EnrichmentData` + typed `MediaMetadata` interface (with `TVMetadata`, `MusicMetadata`, etc.)
   - See [original plan — §3 Layered Media Model](20260318T2119Z-capacitarr-2.0-plan.md#3-layered-media-model)
6. **Build the pluggable enrichment pipeline** in `internal/integrations/enrichment.go` (or a new `internal/enrichment/` package):
   - `Enricher` interface with `Name()`, `Priority()`, `Enrich(items []MediaItem) error`
   - `EnrichmentPipeline` struct with `Run(items []MediaItem)`
   - Migrate existing enrichment logic into individual enricher implementations
   - See [original plan — §4 Pluggable Enrichment Pipeline](20260318T2119Z-capacitarr-2.0-plan.md#4-pluggable-enrichment-pipeline)
7. **Build pluggable scoring factors** in `internal/engine/`:
   - `ScoringFactor` interface with `Name()`, `Key()`, `DefaultWeight()`, `Calculate(item MediaItem) float64`
   - Migrate existing scoring dimensions into individual `ScoringFactor` implementations
   - Register default factors slice
   - See [original plan — §5 Pluggable Scoring Factors](20260318T2119Z-capacitarr-2.0-plan.md#5-pluggable-scoring-factors)
8. **Extract the evaluator from the poller**:
   - `internal/engine/evaluator.go` — reusable scoring + filtering logic
   - Poller becomes an orchestrator calling the evaluator
   - Action Router dispatches results to delete, approval queue, dry-run, or analytics
   - See [original plan — §6 Separate Evaluation from Action](20260318T2119Z-capacitarr-2.0-plan.md#6-separate-evaluation-from-action)
9. **Create the Library abstraction** — `LibraryService` in `internal/services/`:
   - CRUD for `Library` entity (grouping integrations with per-library thresholds)
   - `EffectiveThresholdForIntegration(integrationID)` method
   - See [original plan — §7 Library Abstraction](20260318T2119Z-capacitarr-2.0-plan.md#7-library-abstraction)
10. **Update the poller** (`internal/poller/`) to use the new registry, enrichment pipeline, and evaluator instead of the old manual wiring.
11. **Update all route handlers** to use the refactored services and models. No new endpoints — just update existing ones to compile with the new types.
12. **Update all existing tests** to work with the refactored architecture. Fix compilation errors, update test helpers, adjust assertions.
13. **Run `make ci`** — everything must pass.

### Acceptance Criteria

- [ ] Capability interfaces defined and implemented by all integration clients
- [ ] `IntegrationRegistry` replaces `EnrichmentClients` and switch-statement wiring
- [ ] `overseerr.go` → `seerr.go` rename complete (code, types, constants, tests)
- [ ] `MediaItem` uses layered model (`EnrichmentData`, typed `MediaMetadata`)
- [ ] `EnrichmentPipeline` with individual `Enricher` implementations replaces monolithic `EnrichItems()`
- [ ] `ScoringFactor` interface with individual factor implementations replaces hardcoded scoring loop
- [ ] Evaluator extracted from poller; poller is now an orchestrator
- [ ] `LibraryService` exists with CRUD and `EffectiveThresholdForIntegration()`
- [ ] All existing tests pass with refactored architecture
- [ ] `make ci` passes
- [ ] No new API endpoints or features — behavior is identical to pre-refactor

## Phase 5: Features — Backend APIs

**Goal:** Build the backend services and API endpoints that power the 8 cherry-picked features. The evaluator, enrichment pipeline, and registry from Phase 2 make this mostly "write aggregation logic and expose via endpoints."

**Design reference:** [Original plan — Selected Features](20260318T2119Z-capacitarr-2.0-plan.md#selected-features-8) and [Feature Deep Dives](#appendix-a-feature-deep-dives)

### Steps

1. **Create `AnalyticsService`** in `internal/services/analytics.go`:
   - `GetComposition(libraryID)` — quality, genre, year, type, integration aggregations
   - `GetGrowthTrend(libraryID, period)` — growth over time from historical rollups
   - `GetQualityDistribution(libraryID)` — detailed quality breakdown
   - `GetSizeAnomalies(libraryID)` — bloat detection (items > 2x median for quality profile)
   - Data source: preview cache (existing `MediaItem` slice). No new DB tables for aggregations.
2. **Create `WatchAnalyticsService`** in `internal/services/watch_analytics.go`:
   - `GetDeadContent(libraryID, minAgeDays)` — PlayCount == 0, not on watchlist, added > N days ago
   - `GetStaleContent(libraryID, staleDays)` — LastPlayed > N days, PlayCount > 0, with staleness score
   - `GetPopularity(libraryID)` — heatmap data (genre × year → play count), ranked lists
   - `GetRequestFulfillment(libraryID)` — fulfillment rate, unfulfilled list, per-user breakdown
   - Requires enrichment data; items without enrichment excluded to avoid false positives.
3. **Add API routes:**
   - `GET /api/v1/analytics/composition` → `AnalyticsService.GetComposition`
   - `GET /api/v1/analytics/growth` → `AnalyticsService.GetGrowthTrend`
   - `GET /api/v1/analytics/quality` → `AnalyticsService.GetQualityDistribution`
   - `GET /api/v1/analytics/bloat` → `AnalyticsService.GetSizeAnomalies`
   - `GET /api/v1/analytics/dead-content` → `WatchAnalyticsService.GetDeadContent`
   - `GET /api/v1/analytics/stale-content` → `WatchAnalyticsService.GetStaleContent`
   - `GET /api/v1/analytics/popularity` → `WatchAnalyticsService.GetPopularity`
   - `GET /api/v1/analytics/request-fulfillment` → `WatchAnalyticsService.GetRequestFulfillment`
4. **Add per-integration threshold support to `IntegrationService`:**
   - `UpdateThresholds(integrationID, thresholdPct, targetPct)` method
   - Update `DiskGroupService` to support `EffectiveThresholdForIntegration()`
5. **Add rule impact preview to `RuleService`:**
   - `GetRuleImpact(ruleID)` → count of items affected by this rule
   - Uses the evaluator from Phase 2 to test-run the rule against the preview cache
6. **Add `LibraryService` API routes** (if not already exposed in Phase 2):
   - `GET /api/v1/libraries` — list all libraries
   - `POST /api/v1/libraries` — create library
   - `PUT /api/v1/libraries/:id` — update library
   - `DELETE /api/v1/libraries/:id` — delete library
7. **Publish SSE events** for analytics cache invalidation:
   - `analytics_updated` event when preview cache refreshes
   - Frontend uses this to refetch analytics data
8. **Write tests** for all new services and route handlers:
   - Unit tests for aggregation logic (AnalyticsService, WatchAnalyticsService)
   - Route handler tests using the standard pattern
   - Use in-memory SQLite for service tests
9. **Run `make ci`.**

### Acceptance Criteria

- [ ] All 8 analytics API endpoints return correct JSON responses
- [ ] `AnalyticsService` aggregation logic tested with representative data
- [ ] `WatchAnalyticsService` correctly excludes items without enrichment data
- [ ] Per-integration threshold overrides work in the evaluation loop
- [ ] Rule impact preview returns accurate affected-item counts
- [ ] Library CRUD endpoints functional
- [ ] SSE `analytics_updated` event fires on preview cache refresh
- [ ] All new code has unit tests
- [ ] `make ci` passes

## Phase 7: Migration Script

**Goal:** Build the optional one-way migration that imports 1.x user config into the 2.0 schema.

**Design reference:** [Original plan — Fresh Schema Baseline (Migration UX)](20260318T2119Z-capacitarr-2.0-plan.md#fresh-schema-baseline)

### Steps

1. **Build the migration logic** in `internal/migration/` (new package):
   - Open 1.x database read-only
   - Export: `IntegrationConfig`, `PreferenceSet`, `CustomRule`, `NotificationConfig`, `AuthConfig`
   - Transform: `type = "overseerr"` → `type = "seerr"`; map 1.x fields to 2.0 schema
   - Import into the fresh 2.0 database
   - Skip transient data: approval queue, audit log, engine stats, activity events
2. **Build the CLI command** — `capacitarr migrate --from /config/capacitarr.db`
3. **Build the migration page** frontend:
   - Detect 1.x database on first launch → show migration page instead of login
   - "Welcome to Capacitarr 2.0" header with explanation
   - "Import from 1.x" button → runs migration, shows progress, redirects to login
   - "Start Fresh" button → creates clean 2.0 database, redirects to setup
   - Note about CLI alternative
4. **After migration:** rename 1.x database to `capacitarr.db.v1.bak` (not deleted)
5. **Migration page only appears once** — subsequent launches go to login
6. **Write tests** for transformation logic using fixture 1.x database data
7. **Run `make ci`.**

### Acceptance Criteria

- [ ] Migration correctly imports integrations, rules, preferences, notifications, and auth config

## Appendix A: Feature Deep Dives

Detailed design rationale, implementation notes, and open questions for each feature. Refer to these when implementing the corresponding Phase 5/6 steps.

### Feature 1: Per-Integration Thresholds

**Problem:** Different *arr instances have different storage strategies. Anime Sonarr on a 2TB SSD should trigger at 90%, main Radarr on a 20TB NAS at 80%.

**User-facing:** Each integration card in Settings gets an optional "Override Thresholds" toggle. When enabled, threshold % and target % sliders appear that override the disk group defaults.

**Implementation:**
- Nullable `ThresholdPct` and `TargetPct` columns on `IntegrationConfig`
- `DiskGroup.EffectiveThresholdForIntegration(integrationID)` returns override or default
- Evaluation loop must handle mixed thresholds within a single disk group

**Open question:** The evaluation loop needs to handle mixed thresholds within a single disk group — items from different integrations may have different trigger conditions.

### Feature 2: Library Composition Dashboard

**Problem:** No visibility into what the library contains at a high level. Users know they have 500 movies but not the quality/genre/year distribution.

**Charts:** Quality Profile Distribution (donut), Genre Distribution (horizontal bar), Year Distribution (area chart), Media Type Breakdown (stacked bar by integration), Integration Contribution (treemap), Growth Over Time (line chart from rollups).

**Implementation:** All data from existing `MediaItem` slice in the preview cache. `AnalyticsService` aggregates into chart data structures. API: `GET /api/v1/analytics/composition`. Refreshes via SSE on `preview_updated`.

### Feature 3: Bloat Detection

**Problem:** A 720p movie shouldn't be 45GB. A 4K movie at 2GB is suspicious.

**User-facing:** "Size Anomalies" card showing items > 2x median for quality profile, sorted by worst offenders.

**Implementation:** Group items by quality profile, calculate median + standard deviation, flag outliers. All computation over existing preview cache — zero new API calls. Pure statistical computation.

### Feature 4: Quality Distribution Visualization

**Problem:** No visual answer to "what does my library look like quality-wise?"

**User-facing:** ECharts donut or stacked bar showing 4K / 1080p / 720p / 480p / Unknown by item count and storage. Clicking a segment filters the Library page.

**Implementation:** Group items by `QualityProfile`. Click-to-filter via Library page query parameter (`/library?quality=720p`). `QualityProfile` is already populated for all *arr items.

### Feature 5: Dead Library Items

**Problem:** "I have 500 movies I've never watched and probably never will."

**User-facing:** "Dead Content" report: PlayCount == 0 AND OnWatchlist == false AND AddedAt > N days ago (configurable, default 90). Sorted by size.

**Implementation:** Filter preview cache. Requires enrichment data (media server for play count/watchlist). Items without enrichment excluded to avoid false positives. Needs a configurable "minimum days in library" threshold preference.

### Feature 6: Popularity Heatmap

**Problem:** No visual representation of engagement levels across the library.

**User-facing:** ECharts heatmap (genre × year, color = play count) or bubble chart. Alternative: ranked top/bottom 20 lists.

**Implementation:** Aggregate play counts from preview cache. Requires at least one watch data provider. Recommended default axes: genre × year.

### Feature 7: Stale Content Report

**Problem:** Content that was watched but hasn't been touched in a long time — past its useful life.

**User-facing:** Items where LastPlayed > N days ago (configurable, default 180) AND PlayCount > 0. Sorted by staleness score.

**Implementation:** Staleness score formula: `daysSinceLastPlayed / 365 * (seriesEnded ? 1.5 : 1.0) * (!onWatchlist ? 1.2 : 0.5)`. Configurable staleness threshold preference needed.

### Feature 8: Request Fulfillment Tracking

**Problem:** "Did the person who requested this content actually watch it?"

**User-facing:** "Request Fulfillment" card: fulfillment rate headline stat, unfulfilled requests table, per-user breakdown.

**Implementation:** Filter preview cache where `IsRequested == true`. Split by `WatchedByRequestor`. The enrichment pipeline already cross-references Seerr + watch data.

**Scope limitation:** Binary watched/not-watched is sufficient for 2.0. "Days to watch" metric would require storing request fulfillment date from Seerr (not currently fetched).
